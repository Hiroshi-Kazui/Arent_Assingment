# テストコード実装バグ修正パイプライン

対象スコープ（省略可）: $ARGUMENTS

## このスキルの目的

テストコードの **実装バグ** を修正する。
**テストの期待値・検証内容（何をテストするか）は変更しない。テストの書き方（どう書くか）だけを直す。**

### 修正対象（テストの実装バグ）
- モックの設定不備（未定義、型不一致）
- 非同期処理の待ち方ミス（await 漏れ、タイミング問題）
- セレクタ・ロケータの誤り（E2E）
- import パスの誤り
- テストデータのセットアップ不備

### 修正対象外（仕様・期待値の問題）
- アサーションの期待値変更 → 仕様変更なので `/gen-test-spec` → `/gen-test-code` を再実行
- テストケースの追加・削除 → `/gen-test-spec` → `/gen-test-code` を再実行
- テストが検証するビジネスルールの変更 → 仕様変更

**前提条件**: `/run-test` の結果で失敗が明らかにテストコードの実装バグと判断されていること。

---

## 実行手順

### Step 1: 失敗テストの特定

$ARGUMENTS が指定されている場合:
- スコープに対応するテストファイルのみ対象とする

$ARGUMENTS が未指定の場合:
- 全テストファイルを対象に、まず実行して失敗テストを特定する

```bash
npx vitest run --reporter=verbose 2>&1
npx playwright test --reporter=list 2>&1
```

失敗テストがない場合はレポートを出力して終了する。

**失敗原因の事前判定:**
各失敗テストのエラー内容を確認し、実装バグかどうかを判定する。
- 実装バグ → Step 2 へ
- 期待値・仕様の問題 → 修正対象外としてレポートに記載し、スキップ

---

### Step 2: 修正→再テストループ（最大3回、引数で変更可）

`test-fixer` エージェント（subagent_type）を起動する。

$ARGUMENTS に数値が含まれる場合（例: `issue 5`）、その数値を最大ループ回数とする。
数値がない場合はデフォルト3回。

**プロンプト:**
```
以下の失敗テストを診断し、テストコードの実装バグを修正してください。

## 失敗テスト一覧
{Step 1 で特定した失敗テストのファイルパス、テスト名、エラー出力}

## 重要な制約
- **テストコード（*.test.ts, *.spec.ts）のみ修正可能**
- プロダクションコードは変更禁止
- **テストの期待値・検証内容は変更禁止**（アサーションの値を変えてテストを通すことは禁止）
- 修正できるのはテストの「書き方」のみ（モック設定、非同期処理、セレクタ、import等）

## 修正→再テストループ
最大{N}回の fix → execute → verify ループを実行してください。
- 全テスト合格: ループ終了
- 失敗が残る: 次のループへ
- 最大回数到達: 未解決として報告
- 期待値を変えないと解決できない場合: 実装バグではないため未解決として報告

エージェント定義の診断フローに従い、修正レポートを返却してください。
```

---

### Step 3: 結果レポート

以下をチャットに出力する:

```
## テストコード修正結果

### サマリ
- スコープ: {$ARGUMENTS または「全体」}
- 修正ループ: {実行回数} / {最大回数}
- 修正前: XX failed
- 修正後: YY failed（ZZ resolved）

### 修正内容（実装バグの修正のみ）
| ファイル | 修正内容 | 種別 | 結果 |
|---------|---------|------|------|
| (修正されたテストファイル一覧) | | モック/非同期/セレクタ/import等 | |

### 未解決の失敗（あれば）
| テストファイル | テスト名 | エラー概要 | 未解決理由 |
|--------------|---------|-----------|-----------|
| (最大ループ後も残った失敗) |

### 次のアクション
- 全テスト合格: 完了
- プロダクション側の問題: `/fix-prod` を実行
- 仕様・期待値の問題: `/gen-test-spec` → `/gen-test-code` を再実行
```

---

## 注意事項

- **テストの期待値・検証内容は絶対に変更しない** — テストは仕様の具現化である
- 修正対象は実装バグ（書き方の問題）のみ
- 期待値を変えないと通らない失敗は、プロダクションコード側の問題（`/fix-prod`）または仕様の問題
- ループ回数のデフォルトは3回、引数で変更可能（例: `/fix-test 5`）
- E2E テストの修正・再実行には Docker と dev server が必要
